# Parallelism Benchmarks

## Setup

- **Machine**: Apple Silicon (16 logical CPUs)
- **Build**: `cargo build --release`
- **Tool**: [`hyperfine`](https://github.com/sharkdp/hyperfine) (warmup + 5–10 runs)
- **Data**: synthetic CSV generated by `src/bin/gen.rs` — deposits and withdrawals, 10k unique clients

## Architecture

The engine splits the input file into N equally-sized byte-aligned chunks. N parser threads
read and parse their chunk in parallel (Phase 1), then N processor threads consume the
ordered records via per-shard channels (Phase 2). All transactions for a given client always
land on the same shard, preserving ordering without any locking.

```
file.csv
   │
   ├── chunk 0 ──► parser thread 0 ──► Vec<Record>
   ├── chunk 1 ──► parser thread 1 ──► Vec<Record>     Phase 1 (parallel parse)
   └── chunk N ──► parser thread N ──► Vec<Record>
                                            │
                         route in file order by client_id % N
                                            │
   ├── worker 0  (clients 0, N, 2N, …) ◄──┤
   ├── worker 1  (clients 1, N+1, …)   ◄──┤             Phase 2 (parallel process)
   └── worker N  (clients N-1, …)      ◄──┘
```

## Results

### Wall time by worker count

| Workers |   500k |     1M |   100M |
| ------: | -----: | -----: | -----: |
|       1 | 170 ms | 295 ms | 37.1 s |
|       2 | 250 ms | 390 ms |      — |
|       4 |  62 ms | 119 ms | 12.4 s |
|       8 | 280 ms | 472 ms |  9.8 s |
|      16 | 624 ms | 996 ms |      — |

### Speedup over 1 worker

| Workers | 500k |   1M | 100M |
| ------: | ---: | ---: | ---: |
|       2 | 0.7× | 0.8× |    — |
|       4 | 2.4× | 2.5× | 3.0× |
|       8 | 0.6× | 0.6× | 3.8× |
|      16 | 0.3× | 0.3× |    — |

## Key Findings

**1. The bottleneck shifts with data volume.**

At 500k–1M records, 4 workers is optimal. At 100M records, 8 workers pulls ahead.
Thread startup, join overhead, and channel contention cost the same regardless of file
size — at small inputs those fixed costs dominate; at large inputs the parallelism savings
outweigh them.

**2. `available_parallelism()` is the wrong default.**

Using all 16 logical CPUs is consistently the worst outcome. At 1M records it is
**3.4× slower** than a single thread. System time (context switching overhead) scales
roughly linearly with worker count when chunks are small.

**3. User time reveals true parallel efficiency.**

At 100M records with 8 workers: 68s User time / 8 cores ≈ 8.5s ideal vs 9.8s actual.
That <15% overhead shows the architecture is close to linearly scalable once chunk size
is large enough to amortise coordination costs.

**4. The serial routing phase is the remaining bottleneck.**

Phase 1 (parsing) is fully parallel. Phase 2 routing — iterating the ordered Vecs and
sending to worker channels — is still single-threaded. At very large scale this becomes
the next thing to optimise (e.g. parallel routing with per-chunk sequence numbers, or
pre-partitioned input files as in a Kafka-style architecture).

## Reproducing the benchmarks

### 1. Build

```bash
cargo build --release
```

### 2. Generate test data

```bash
# Usage: gen <num_records> [num_clients]
./target/release/gen 500000    > /tmp/payments_500k.csv
./target/release/gen 1000000   > /tmp/payments_1m.csv
./target/release/gen 100000000 > /tmp/payments_100m.csv  # ~5s, produces a 2.9 GB file
```

### 3. Install hyperfine

```bash
brew install hyperfine
```

### 4. Run

```bash
hyperfine \
  --warmup 3 --runs 10 \
  './target/release/octo-payments /tmp/payments_1m.csv 1' \
  './target/release/octo-payments /tmp/payments_1m.csv 2' \
  './target/release/octo-payments /tmp/payments_1m.csv 4' \
  './target/release/octo-payments /tmp/payments_1m.csv 8'
```

For the 100M file, reduce runs to avoid long waits:

```bash
hyperfine \
  --warmup 1 --runs 5 \
  './target/release/octo-payments /tmp/payments_100m.csv 1' \
  './target/release/octo-payments /tmp/payments_100m.csv 4' \
  './target/release/octo-payments /tmp/payments_100m.csv 8'
```

### Running the engine directly

```bash
# explicit worker count
./target/release/octo-payments transactions.csv 4

# auto worker count (available_parallelism — not recommended, see findings above)
./target/release/octo-payments transactions.csv
```
